# 《边缘世界 (RimWorld)》 AI 叙事者模组 —— 项目规划书

## 1. 项目概述

本项目为《边缘世界》开发一个由大语言模型驱动的智能叙事者模组。传统叙事者（Cassandra、Phoebe、Randy）依赖固定概率和随机数触发事件，叙事逻辑缺乏对殖民地整体状态的语义理解。本模组通过定期抓取游戏实时状态数据发送给云端 LLM，让 LLM 充当"跑团 DM"角色，提前为未来 N 天规划事件队列，实现有上下文感知的、连贯的动态叙事。

**核心差异化特性：**

- LLM 一次性规划未来多天的事件序列，而非逐事件即时决策
- 事件按精确 tick 排程，与游戏时间系统无缝对接
- 类 TCP 协议的可靠请求-响应协议，应对网络不稳定
- 多层输出校验机制，确保 LLM 回复合法可用
- 失败时平滑降级到原版叙事者，不破坏游戏体验

## 3. 核心机制

### 3.1 类 TCP 协议滑动窗口调度算法

这是本模组最核心的机制。设计参考 TCP 协议的确认应答（ACK）和超时重传机制。

#### 3.1.1 参数定义

| 参数     | 符号      | 类型               | 说明                                    |
| ------ | ------- | ---------------- | ------------------------------------- |
| 缓冲期    | `B`     | int（游戏天数转 ticks） | 例如 2 天 = 120000 ticks。提前量，确保事件到队列时已缓冲 |
| 规划窗口   | `N`     | int（游戏天数转 ticks） | LLM 一次性规划的天数，例如 5 天 = 300000 ticks    |
| 超时时间   | `T`     | float（现实秒）       | 例如 60s。请求发出后的等待上限                     |
| 最大重传次数 | `R_max` | int              | 超时后允许重发的次数                            |
| ACK    | `ack`   | int（tick）        | 已被 LLM 有效回复覆盖到的最晚 tick。初始值为 0         |

以上参数除 `ack` 为运行时状态外，其余均可通过 Mod 配置界面调整。

#### 3.1.2 状态机

```
                    ┌─────────────────────────────┐
                    │        空闲状态 （Unlocked）    │
                    │ 条件: ack - curTick >= B     │
                    │ 动作: 无                     │
                    └────────────┬────────────────┘
                                 │ ack - curTick < B
                                 ▼
                    ┌─────────────────────────────┐
                    │        请求中 （Locked）       │
                    │ 上锁 → 发送请求 → 启动计时器    │
                    │ 剩余重传次数 = R_max          │
                    │ expected_range = [ack+1,     │
                    │   ack+1+N*60000-1]          │
                    └────────────┬────────────────┘
                                 │
               ┌─────────────────┼──────────────────┐
               ▼                 ▼                   ▼
        收到有效回复          计时器超时          curTick 进入窗口首日
        (from_tick         (剩余重传 > 0)      (缓冲耗尽)
         == ack+1)              │                   │
               │                ▼                   ▼
               ▼          重传(剩余-1)         停止请求
        解锁 + 关计时器     重启计时器         弹窗通知玩家
        ack = to_tick       保持 Locked       ┌────┴────┐
        事件排入队列                        [重试]    [放弃]
        回到 Unlocked                        │         │
                                       忽略B,      原版接管
                                       立即请求    永久降级标记
                                       下一窗口
```

#### 3.1.3 算法伪代码

```
OnGameTick(curTick):
    if 永久降级标记 == true:
        return  // 原版叙事者全权接管

    if 请求锁 == Unlocked and ack - curTick < B:
        expected_start = ack + 1
        expected_end = ack + N * 60000
        snapshot = SnapshotCollector.Collect()
        request = PromptBuilder.BuildRequest(snapshot, expected_start, expected_end)
        AIChatServiceAsync.Send(request, OnResponseReceived)
        上锁()
        启动计时器(T秒, 暂停时冻结)
        重传剩余 = R_max

OnTimerExpired():
    if 重传剩余 > 0 and curTick < expected_start:
        重传剩余 -= 1
        AIChatServiceAsync.Send(相同请求, OnResponseReceived)  // 重传
        重启计时器(T秒)
    else:
        触发降级()  // 缓冲耗尽

OnResponseReceived(response):
    parsed = ResponseParser.Parse(response)

    // 第1-3层 Guard 不通过 → 追加修正消息，重新发送（不是重传，是新的请求）
    if parsed.needs_correction:
        修正消息 = FormatCorrectionMessages.Build(parsed.violation)
        AIChatServiceAsync.Send(修正消息, OnResponseReceived)
        return  // 锁保持，计时器继续倒计时

    // ACK 校验
    if parsed.plan_range.from_tick != ack + 1:
        丢弃(response)  // 这是过期回复，忽略
        return

    // 有效回复
    解锁() + 关闭计时器()
    过滤掉 parsed.events 中 scheduled_tick < curTick 的条目 // 已过期的丢弃
    剩余事件 → EventQueue.InsertAll(parsed.events)  // 按 tick 升序排列
    ack = parsed.plan_range.to_tick  // ACK 推进
    // 注意: ack更新 和 事件入队 必须在同一帧内完成（原子性）

触发降级():  // curTick >= expected_start，仍未收到有效回复
    停止请求 + 解锁()
    显示游戏内弹窗:
        "AI 叙事者连接失败（请求超时）。
        窗口 [Day X 至 Day Y] 已错过，原版叙事者已接管。
        你可以选择："
        [重试] → 忽略B, 立即请求 ack+1 ~ ack+1+N*60000
        [放弃] → 永久降级标记 = true（可通过开发者按钮或管理面板手动恢复）
```

#### 3.1.4 具体时间轴示例

```
配置: B=2天(120000tick), N=5天(300000tick), T=60s, R_max=2
初始: ack=0, curTick=0

Tick 0 (Day 0):
  ack(0) - curTick(0) = 0 < 120000 ✓ → 发送请求
  请求范围: tick 120000 ~ tick 419999 (Day 2~6)
  上锁, 计时器=60s, 重传剩余=2

场景A —— 正常:
  30s后收到回复，plan_range: from=120000, to=419999 ✓
  解锁, ack = 419999, 事件排入队列
  ────────────────────────────────────
  Tick 300000 (Day 5):
    ack(419999) - curTick(300000) = 119999 < 120000 ✓ → 发送下一请求
    请求范围: tick 420000 ~ tick 719999 (Day 7~11)

场景B —— 超时重传 → 成功:
  60s后计时器超时, 重传剩余: 2→1, 重传
  再60s后超时, 重传剩余: 1→0, 再重传
  10s后收到回复, from=120000 ✓
  解锁, ack=419999, 排入队列

场景C —— 缓冲耗尽 → 降级:
  60s→重传(1) → 60s→重传(0) → tick已到120000 (Day 2开始)
  显示弹窗: 玩家选[重试] → 忽略B, 立即请求 tick 420000~719999
           玩家选[放弃] → 永久降级标记=true, 原版接管
```

#### 3.1.5 请求锁与计时器细节

- **锁的范围**：任何时刻最多只能有一个请求处于飞行中。这是防止并发请求的核心手段。
- **计时器时间域**：使用现实时间（`Time.realtimeSinceStartup` 差值），而非游戏 tick。因为 LLM 的延迟取决于网络状况，与游戏速度无关。
- **暂停时冻结**：当游戏暂停（`Find.TickManager.Paused`）时，计时器暂停计时。否则玩家暂停去吃饭回来后会发现大量超时。
- **ACK 作为请求校验**：请求发送时记录 `expected_from_tick = ack + 1`。回复到达时校验 `response.from_tick == expected_from_tick`。如果 LLM 回复覆盖范围不是从 `ack+1` 开始（例如从 `ack+1000` 开始），说明它误解了请求范围，应丢弃此回复。

#### 3.1.6 永久降级的恢复

`permanentDegraded == true` 后，玩家并非永远无法恢复，而是提供两种恢复途径：

**途径一：开发者模式（开发阶段使用）**

Settings 中留一个调试按钮，直接调用 `StoryMakerState.ResetDegradation()`。此按钮仅在开启开发者模式时可见。

**途径二：游戏底部菜单管理面板（正式版本）**

在 RimWorld 游戏屏幕底部菜单栏（World / History / Factions 等选项卡区域）增加一个 StoryMaker 选项卡。点击后打开管理面板，面板显示：

| 信息项        | 说明                    |
| ---------- | --------------------- |
| 连接状态       | 正常 / 降级中 / 永久降级 / 请求中 |
| ACK 位置     | 当前 ack 对应的 Day 数      |
| 事件队列长度     | 队列中排程了多少个事件           |
| 最近一次请求时间   | 游戏时间                  |
| 最近一次错误     | 错误类型与时间               |
| Token 消耗统计 | 本次会话累计                |

面板底部提供操作按钮：

| 按钮         | 功能                              | 适用场景                   |
| ---------- | ------------------------------- | ---------------------- |
| **恢复连接**   | 清除永久降级标记，立即发送下一窗口请求             | 玩家解决了网络/API Key 问题后恢复  |
| **跳过当前窗口** | 丢弃飞行中的请求（如有），ack 不变，重新评估是否需要发请求 | 当前 LLM 响应异常缓慢，想主动放弃本窗口 |
| **立即刷新**   | 忽略缓冲期，立即请求从 ack+1 开始的窗口         | 读档后发现 ack 太旧，手动触发更新    |
| **重置统计数据** | 清空 empty plan 计数、token 统计等      | 调试用途                   |

> **设计原则：** 管理面板不应让玩家直接查看或修改底层数据（如直接改 ack 值），只提供预设的操作按钮。这样既给了玩家控制力，又防止误操作。

#### 3.1.7 StoryMaker StorytellerDef 与 Harmony Patch 策略

**创建新 StorytellerDef：**

新建 `Storyteller_StoryMaker`，`ParentName="BaseStoryteller"`，继承 BaseStoryteller 的全部 comps（通关任务、深钻井虫害、黑衣人救援等游戏必需事件）。子类的 comps 最小化——不添加任何自动生成事件的 comp，所有叙事事件由 LLM 驱动。玩家在开局选叙事者时从列表中选择 "AI 叙事者"。

**Caravan/Temp Map 事件交由原版：**

远行队和临时地图事件（`CategoryIndividualMTBByBiome`——Misc / ThreatSmall / ThreatBig）在 StoryMaker StorytellerDef 中显式保留，由原版自动生成，LLM 不干预这些事件。

**Tick 注入——Harmony Prefix Patch：**

采用 Scheme A：Harmony prefix patch `Storyteller.StorytellerTick()`，在方法开头判断 `Find.Storyteller.def.defName == "StoryMaker"`，若匹配则调用 `EventScheduler.OnTick(curTick)`。原版方法体继续执行——`IncidentQueueTick()` 处理继承自 BaseStoryteller 的触发式事件（如 `StrangerInBlackJoin`），`MakeIncidentsForInterval()` 因无自动发射 comp 而返回空列表。

```
[HarmonyPatch(typeof(Storyteller), "StorytellerTick")]
static class Patch_StorytellerTick
{
    static void Prefix()
    {
        if (Find.Storyteller?.def?.defName == "StoryMaker")
            EventScheduler.OnTick(Find.TickManager.TicksGame);
    }
}
```

约 15 行代码。StorytellerTick 不是热门 patch 点，冲突风险极低。

### 3.2 协议数据格式

#### 3.2.1 请求消息结构（模组 → LLM）

每次请求由**三段内容**组成，按顺序拼接为一条 `user` 角色的消息：

**第1段：系统提示词（System Prompt）**

在对话开始时通过 `system` 角色消息发送，不在每条 user 消息中重复。内容包括：

- LLM 身份声明（"你是 RimWorld 的 AI 叙事者"）
- 游戏数据各字段的含义说明
- 回复 JSON 各字段的含义及合法值范围
- 支持的事件类型列表及每个事件类型的参数说明

**第2段：玩家个性化定制**

玩家在 Mod 配置中通过自然语言描述自己希望叙事者具备的性格和风格。此内容在每个 user 请求的开头拼接。示例：

> 你是一个仁慈但有原则的叙事者。你喜欢在平静期给玩家惊喜，但不会恶意摧毁玩家的基地。你的叙事风格偏向沙盒文明模拟，偶尔有黑色幽默。

**第3段：规范化游戏数据（JSON）**

游戏状态快照以 JSON 格式呈现，使用 markdown 代码块（` ```json ... ``` `）包裹，方便 LLM 区分数值与自然语言指令。

**快照字段清单（已确认）：**

| 类别   | 字段                                                                      | 说明                                                                                  |
| ---- | ----------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| 殖民地  | `name`, `population`, `average_mood` (0~1), `food_days`, `total_wealth` | 核心叙事输入                                                                              |
| 环境   | `season`, `biome`, `current_temperature`                                | 温度对规划有意义；天气瞬息万变不需要                                                                  |
| 派系   | `faction_relations[]`：每项 `name` + `relation` (-100~100)                 | LLM 选择袭击/商队来源                                                                       |
| 近期事件 | `recent_events[]`：每项 `type`（事件名）+ `category`（类别）                        | 通过 Harmony patch `Storyteller.TryFire()` 实时捕获，维护 Incident 白名单栈。排除远行队和 Special 一次性事件 |
| 近期死亡 | `recent_deaths[]`：每项 `pawn_name`（角色名）                                   | 通过 Harmony patch `LetterStack.ReceiveLetter()` 仅捕获 `LetterDefOf.Death` 信件           |
| 时间   | `current_tick`, `request_range` (from_tick / to_tick)                   | LLM 需感知游戏进度                                                                         |

#### 3.2.3 外接系统提示词注入（Mod 兼容）

### 3.3 四层 Guard 校验机制

LLM 的回复不可信——格式可能错误、字段可能缺失、值可能不合规。每一层 Guard 如果检测到违规：

1. 将违规标签（如 `PARSE_ERROR`）和修正指示拼成一条新的 user 消息
2. 将这条消息追加到当前对话历史中（替换被拒绝的 assistant 回复）
3. 向 LLM 重新发送完整的修正后对话
4. 请求锁保持锁定，计时器继续倒计时

每层 Guard 的重试上限为 **1 次**。达到上限后跳过该层（降级接受，或触发最终失败）。

#### 第1层：Parse Guard —— 验证回复是否为合法 JSON

**检查项目：**

- 回复是否为空字符串
- 回复是否能解析为 JSON（`JsonUtility.FromJson` 或手动解析）
- LLM 是否在 JSON 之外包裹了 markdown 代码块标记（` ```json ... ``` `）——先尝试剥离再解析

**违规时追加的消息：**

```
STORYMAKER_PARSE_ERROR
你的上一条回复不是合法的 JSON 格式。请严格输出一个 JSON 对象，
首字符 { 末字符 }，不要附加任何 markdown、自然语言说明或代码块标记。
你应当回复的 JSON 结构示例：
{
  "plan_range": { "from_tick": <整数>, "to_tick": <整数> },
  "empty_plan": false,
  "narrative_summary": "<本窗口叙事意图简述>",
  "events": [ ... ]
}
```

#### 第2层：Schema Guard —— 验证必需字段是否存在

**检查项目：**

- `plan_range` 是否存在，是否包含 `from_tick` 和 `to_tick` 且均为正整数
- `events` 是否为数组（即使为空数组也应是 `[]`）
- `empty_plan` 是否为 bool
- `narrative_summary` 是否为非空字符串

**违规时追加的消息：**

```
STORYMAKER_SCHEMA_VIOLATION=<具体缺失字段标签>
你的上一条回复缺少必要字段或字段类型不正确。请修正后重新输出完整的 JSON 对象。
缺失或类型错误的字段: <列出具体项>
```

常见 violation 标签：`missing_plan_range`、`missing_events_array`、`missing_empty_plan`、`missing_narrative_summary`、`invalid_tick_type`。

#### 第3层：Event Guard —— 验证事件条目合法性

**检查项目（对 `events` 数组中的每一项）：**

- `event_id` 是否为非空字符串
- `scheduled_tick` 是否为正整数且在 `plan_range.from_tick ~ plan_range.to_tick` 范围内
- `event_type` 是否在 `ActionRegistry` 支持的事件类型列表中
- `parameters` 是否为有效字典
- `narration_text` 如果存在，必须是非空字符串（该字段为可选，LLM 可省略）

**违规时追加的消息：**

```
STORYMAKER_EVENT_VIOLATION=<具体违规标签>, event_index=<i>
事件列表中第 <i> 个事件存在问题: <具体描述>
请修正该事件条目。支持的事件类型: <列出 ActionRegistry 中注册的事件类型>
每个事件必须包含: event_id, scheduled_tick(在plan_range范围内), event_type, parameters。
可选字段: narration_text(叙事者口吻的一句话，用于事件触发时显示/配音)。
```

#### 第4层：Empty Plan Guard —— 连续空事件检测

**与前三层不同：此 Guard 不触发重试，而是修改下一轮的 prompt。**

- 维护全局计数器 `consecutive_empty_plans`
- 每次 `empty_plan == true` 且 `events` 为空数组 → `consecutive_empty_plans += 1`
- 否则 → `consecutive_empty_plans = 0`
- 当 `consecutive_empty_plans >= 3` 时，下一轮的 system prompt 追加：

```
警告：你已经连续 3 个规划窗口没有安排任何事件。
本窗口你**必须**至少安排 1 个合理事件。如果确实无事可发生，
请在 narrative_summary 中给出充分的叙事理由。
```

- 如果追加了上述警告后，LLM 仍然返回 `empty_plan == true` 且无事件 → 丢弃该回复，**再追加一次更严厉的修正消息重新发送**（这一次允许一次重试）。

### 3.4 事件调度与执行

#### 3.4.1 事件数据模型

```csharp
public class ScheduledEvent {
    public string event_id;
    public int scheduled_tick;
    public int generated_at_world_tick;  // LLM 生成此事件时的世界 tick，用于读档过期检测
    public string event_type;
    public Dictionary<string, object> parameters;
    public float intensity_multiplier;   // 威胁类事件的 LLM 强度乘数（0.5~2.0），非威胁事件忽略
    public string narration_text;        // 叙事者口吻的一句话（可选），事件触发时显示/配音
    public string narrative_context;     // 该事件在叙事中的意义（LLM 内部参考）
}
```

`narration_text` 字段：

- 由 LLM 在 system prompt 中被要求附带
- 即使不安装 TTS mod，StoryMaker 也会在事件触发时将此文本显示为 Letter 或消息
- 若安装了 TTS mod，TTS mod 在事件触发前将此文本发送给 TTS API 生成语音

#### 3.4.2 事件队列

数据结构：按 `scheduled_tick` 升序排列的优先队列。

```csharp
class EventQueue {
    List<ScheduledEvent> events;  // 始终保持升序排列

    void InsertAll(List<ScheduledEvent> newEvents); // 插入并排序
    ScheduledEvent Peek();        // 查看队首（最近即将触发的事件）
    ScheduledEvent Pop();         // 弹出队首
    void Serialize(ExposeData);   // 存档序列化
}
```

每个 Tick 检查：`if (EventQueue.Peek().scheduled_tick <= curTick) { EventQueue.Pop() → ActionExecutor.Execute() }`

#### 3.4.3 事件生命周期钩子

为支持外部 mod（如 TTS mod）在事件执行前后注入逻辑，`ActionExecutor` 提供以下钩子：

```csharp
public static class ActionExecutor {
    // 外部 mod 可注册这两个委托
    public static Func<ScheduledEvent, bool> OnEventWillExecute;  // 返回 false 可阻止执行
    public static Action<ScheduledEvent> OnEventExecuted;
    public static Action<ScheduledEvent> OnEventExecutionFailed;

    public static bool Execute(ScheduledEvent evt, EventExecutionMode mode) {
        // 触发前钩子
        bool? hookResult = OnEventWillExecute?.Invoke(evt);
        if (hookResult == false) return false;  // 钩子阻止

        // 在事件触发时将 narration_text 显示为 Letter
        if (!string.IsNullOrEmpty(evt.narration_text)) {
            DisplayNarrationLetter(evt);
        }

        // 执行事件
        bool success = DispatchAndExecute(evt, mode);

        // 触发后钩子
        if (success) OnEventExecuted?.Invoke(evt);
        else OnEventExecutionFailed?.Invoke(evt);

        return success;
    }
}
```

**TTS mod 使用钩子的流程：**

1. 注册 `OnEventWillExecute`，在其中将事件的 `narration_text` 发送 TTS API
2. 如果音频已就绪，返回 `true`（允许执行），钩子内同步播放音频
3. 如果音频尚未就绪（TTS 还没返回），返回 `true`（允许执行但静默跳过语音）——不阻止事件触发
4. 注册 `OnEventExecuted`，在事件执行后释放该事件对应的音频内存

#### 3.4.4 事件类型注册表

`ActionRegistry` 是一个静态注册表，映射 `event_type` 字符串到执行函数：

public static class ActionRegistry {

### 3.5 异步通信机制

基于 RimChat 验证过的 `MonoBehaviour + UnityWebRequest` 模式。

**实现要点：**

1. 创建一个 `GameObject("StoryMakerAsyncService")` 并挂载 `DontDestroyOnLoad`。
2. 该 GameObject 上的 `MonoBehaviour.Update()` 每帧轮询 `UnityWebRequest.isDone`。
3. 网络完成时，将结果投递到主线程安全回调队列，在 `Update()` 中统一执行回调。
4. 请求锁在回调执行时更新（上锁/解锁），保证回调在游戏主线程执行。

**为什么不用 C# 原生 async/await？**

- RimWorld 运行在 Unity 的 Mono 运行时上，不完全支持现代 C# 的 `Task` 并行机制
- `UnityWebRequest` 的回调模式已在 RimChat 中验证可行
- 主线程安全由 `Update()` 回调队列保证，避免多线程竞态

#### 3.5.1 HTTP 层错误处理

借鉴 RimChat 的 `ResolveFailureReason` / `FormatProtocolError` / `LooksLikeTimeout` 体系，对 `UnityWebRequest.result` 的非 Success 状态分类处理：

| 错误类型           | `failure_reason` 标签     | 行为                                         |
| -------------- | ----------------------- | ------------------------------------------ |
| 超时             | `timeout`               | 触发 TCP 层重传（消耗 R_max）                       |
| 连接失败（DNS/网络断开） | `connection_error`      | 触发 TCP 层重传                                 |
| HTTP 401 认证失败  | `http_401`              | **不重传**，弹窗提示玩家检查 API Key                   |
| HTTP 404 端点不存在 | `http_404`              | **不重传**，弹窗提示玩家检查 URL                       |
| HTTP 429 限流    | `http_429`              | HTTP 层冷却后重试 1 次（不消耗 R_max）；若仍失败则触发 TCP 层重传 |
| HTTP 5xx 服务端错误 | `http_5xx`              | HTTP 层冷却后重试 1 次（不消耗 R_max）；若仍失败则触发 TCP 层重传 |
| 响应数据解析失败       | `data_processing_error` | 同 Parse Guard 违规——发送修正请求                   |
| 响应体为空          | `empty_response`        | 触发 TCP 层重传                                 |
| 其他未预期 HTTP 错误  | `unexpected_http_error` | 触发 TCP 层重传                                 |

**双层重试机制：**

```
HTTP 层（协程内）
  ├─ 5xx / 429: 短暂冷却后重试 1 次
  ├─ 401 / 404: 直接失败（不重试，弹窗提示）
  └─ 其余: 直接失败
        ↓ 协程返回 error
TCP 协议层（EventScheduler）
  └─ 计时器超时或 HTTP 层失败 → 消耗 R_max → 重传完整请求
```

两层分工明确：HTTP 层处理瞬时服务端错误（快速重试不计入 R_max），TCP 协议层处理整请求超时/失败。

**上下文变化保护：**

`AIChatServiceAsync` 维护一个 `contextVersion` 计数器。每次存档/读档/新游戏时版本递增，所有飞行中请求的协程在收到 HTTP 回调时校验版本——若版本已变则丢弃回调，防止旧存档的回调污染新存档状态。

### 3.6 存档序列化

#### 3.6.1 需要序列化的状态

| 字段                         | 类型                     | 说明                        |
| -------------------------- | ---------------------- | ------------------------- |
| `ack`                      | `int`                  | 当前 ACK 指针                 |
| `eventQueue`               | `List<ScheduledEvent>` | 当前事件队列（所有未触发的排程）          |
| `lastScheduledTick`        | `int`                  | 最后一次发出的请求的 `expected_end` |
| `consecutiveEmptyPlans`    | `int`                  | 连续空事件计数                   |
| `permanentDegraded`        | `bool`                 | 是否已永久降级（含降级原因）            |
| `pendingCorrectionRequest` | `bool`                 | 是否有尚未完成的格式修正请求（可选，存档时可丢弃） |

#### 3.6.2 不需要序列化的状态

| 字段           | 原因                                         |
| ------------ | ------------------------------------------ |
| 飞行中的 HTTP 请求 | 存档时直接丢弃——读档后会根据 ack 和 curTick 的关系自动触发新请求   |
| HTTP 响应回调    | Unity 对象不可序列化                              |
| 请求锁状态        | 读档后默认为 Unlocked，由 `OnGameTick` 重新判断是否需要发请求 |

#### 3.6.3 原子性保证

`ack` 的更新和 `eventQueue` 的插入必须在**同一帧**内完成。不在同一帧之间保存中途状态，因此存档中的 ack 和 eventQueue 始终保持一致。

#### 3.6.4 读档行为

读档后 `OnGameLoad()` 执行：

1. 恢复 ack、eventQueue、permanentDegraded
2. 如果 `permanentDegraded == true`：停止，原版接管
3. 切换 `ack - curTick < B`：`OnGameTick` 将自动触发新请求
4. 过滤 eventQueue 中 `scheduled_tick < curTick` 的过去事件（立即丢弃并执行或直接丢弃）
5. 对队列中事件生成时间与当前时间差距过大的事件（`generated_at_world_tick` 与 curTick 差值超过阈值），标记为"过期"——可选地在下一次 LLM 请求中作为 deviation_report 告知 LLM 重新评估

### 3.9 交互式对话功能（Phase 5+，预留接口）

此功能允许玩家在游戏过程中与 AI 叙事者进行**自由文本对话**，叙事者可以**绕过事件队列，即时触发一次性事件**。

#### 3.9.1 功能场景

玩家经历了一场由叙事者安排的精彩袭击后，通过游戏内对话框向叙事者发送消息：

> 玩家："刚才那场袭击太精彩了，我喜欢你的风格！"

LLM 叙事者回复：

> AI："很高兴你喜欢。作为奖励，我会让一支帝国商队顺路经过你的殖民地。保持警惕，更多的挑战还在前方。"

同时，LLM 的回复中附带一个即时事件标记，模组解析后**立刻**触发 `TraderArrival` 事件，不排入事件队列。

#### 3.9.2 TTS 与对话延迟

交互式对话中，LLM 的回复可能包含两部分：`dialogue_text`（对话文本）和 `immediate_event`（即时事件）。如果安装了 TTS mod，这三件事必须在同一时刻发生：**文字显示、语音播放、事件执行**。

处理流程：

```
收到 LLM 对话回复
    ↓
解析出 dialogue_text + immediate_event（可选）
    ↓
通知 TTS mod（如果已安装）：发送 dialogue_text → TTS API
    ↓
    ├── TTS 已安装:
    │       等待 TTS 音频返回
    │         ↓
    │       ┌─ 音频就绪 ──────────────────────┐
    │       │  同步执行：                      │
    │       │  · 显示 dialogue_text           │
    │       │  · 播放音频                      │
    │       │  · 触发 immediate_event（如有）   │
    │       └────────────────────────────────┘
    │
    └── TTS 未安装:
           直接：显示 dialogue_text + 触发 immediate_event（如有）
```

关键点：**即时事件必须等待音频就绪才能执行**，不允许"事件先触发、语音后播放"或"先显示文字、后放语音"。这与事件队列中的 `narration_text` 策略不同——队列事件是按 tick 自动触发的，等不了；对话中的即时事件是玩家主动发起的，可以等。

为此，`DialogueHandler` 预留 TTS 钩子：

```csharp
public static class DialogueHandler {
    // TTS mod 注册此委托以注入语音生成
    // 返回的 Task 完成时表示音频已就绪
    public static Func<string, Task<AudioClip>> OnGenerateTTS;

    // 对话回复完全就绪时的回调（文本 + 音频 + 事件统一触发点）
    // TTS mod 调用此回调以通知 StoryMaker: 可以显示了
    public static Action<DialogueResponse> OnDialogueReady;
}
```

**两种场景的策略对比：**

| 场景                      | 语音未就绪时的行为      | 原因                        |
| ----------------------- | -------------- | ------------------------- |
| 事件队列的 `narration_text`  | 跳过语音，事件按时触发    | 事件按 tick 调度，不能延迟          |
| 交互对话的 `dialogue_text`   | 延迟文本显示，等音频     | 文本是对话体验核心，但等待由 TTS mod 负责 |
| 交互对话的 `immediate_event` | **延迟事件执行**，等音频 | 必须文字+语音+事件三者同步触发          |

#### 3.9.3 与调度请求的区别

|          | 调度请求（3.1）                      | 交互对话（本功能）                  |
| -------- | ------------------------------ | -------------------------- |
| **触发时机** | 自动，由 ack - curTick < B 触发      | 玩家主动发起                     |
| **数据量**  | 完整游戏状态快照                       | 精简版数据（仅基本殖民地概况）            |
| **回复格式** | 完整 JSON（plan_range + events[]） | 轻量 JSON（可选的即时事件 + 叙事者对话文本） |
| **事件排程** | 排入 EventQueue，按 tick 触发        | 即时执行，绕过队列                  |
| **并发控制** | 请求锁互斥，同时只允许一个飞行请求              | 可以在调度请求飞行中进行（独立的轻量请求）      |

#### 3.9.4 交互对话的回复格式（轻量版）

```json
{
  "dialogue_text": "很高兴你喜欢。作为奖励，我会让一支帝国商队顺路经过你的殖民地。保持警惕，更多的挑战还在前方。",
  "immediate_event": {
    "event_type": "TraderArrival",
    "parameters": {
      "faction": "帝国",
      "trader_kind": "BulkGoods"
    },
    "narrative_context": "叙事者因玩家表扬而主动给予的奖励事件"
  }
}
```

`immediate_event` 为可选字段。当为 `null` 或缺失时，仅显示对话文本而不触发事件。

#### 3.9.5 即时事件的安全限制

为避免玩家通过反复对话"刷"事件，执行层实施以下限制：

| 限制项         | 规则                                           |
| ----------- | -------------------------------------------- |
| **频率限制**    | 两次对话触发的即时事件间隔至少 N 游戏 ticks（可配置），使用游戏时间而非现实时间 |
| **事件类型白名单** | 即时事件默认仅允许**非破坏性**事件（商队、访客、物资空投等），禁用致命袭击      |
| **事件强度上限**  | 即时事件的 `points` 默认不超过当前殖民地财富对应的 50%（比正常事件弱）   |
| **全局冷却**    | 每个调度窗口内最多触发 1 次即时事件                          |

**"无限制模式"开关：** Mod 配置中提供一个名为"允许叙事者通过对话发动任意事件"的开关。此开关开启后：

- 白名单/黑名单检查全部跳过，任何事件类型均可即时触发
- 强度上限取消，事件 `points` 按正常计算
- 频率限制和全局冷却仍然生效（防止刷事件）

> 部分玩家可能希望叙事者能通过对话即时发动袭击等破坏性事件，此开关满足该需求。默认关闭以保证安全。即使开关开启，频率限制和全局冷却依然作为最后的保护层。

#### 3.9.6 架构预留

`Dialogue/` 模块在 Phase 1-4 不实现，但需预留接口：

```csharp

// IActionHandler 接口预留

    bool Execute(ScheduledEvent evt, EventExecutionMode mode);
    bool IsAllowedInImmediateMode { get; }  // 即时模式白名单

}
```

### 3.10 外部 TTS 模组接入预留

TTS（文本转语音）功能作为独立 mod 发布，不在 StoryMaker 内部实现。但 StoryMaker 为此预留以下接口和钩子。

#### 3.10.1 接入点总览

| 接入点                                 | 位置              | TTS mod 用途                                |
| ----------------------------------- | --------------- | ----------------------------------------- |
| `ScheduledEvent.narration_text`     | 事件数据模型（3.4.1）   | TTS mod 读取此字段发送 TTS API，生成语音              |
| `ActionExecutor.OnEventWillExecute` | 事件生命周期钩子（3.4.3） | 事件触发前播放语音；音频未就绪时静默跳过                      |
| `ActionExecutor.OnEventExecuted`    | 事件生命周期钩子（3.4.3） | 事件触发后释放该事件的音频内存                           |
| `DialogueHandler.OnGenerateTTS`     | 对话处理器（3.9.2）    | 对话文本的 TTS 语音生成（返回 AudioClip 的异步任务）        |
| `DialogueHandler.OnDialogueReady`   | 对话处理器（3.9.2）    | TTS mod 调用此回调通知 StoryMaker 文字+音频+事件三者同步触发 |
| `PromptInjector`                    | 提示词注入（3.2.3）    | TTS mod 注入要求 LLM 为每个事件附带 `narration_text` |

#### 3.10.2 Prompt 注入示例

TTS mod 在 `StaticConstructorOnStartup` 中调用：

```csharp
PromptInjector.Register(new PromptInjectionEntry {
    ModName = "StoryMaker TTS",
    Category = "output_format",
    Content = "在 events 数组的每个事件对象中，必须包含 narration_text 字段。"
            + "用叙事者的口吻写一句话（中文），用于事件触发时向玩家语音播报。"
            + "例如：\"我看你最近过得挺潇洒，看看这个袭击你能不能接得住。\""
});
```

#### 3.10.3 TTS mod 职责范围

TTS mod 独立负责：

- TTS API 的调用与认证（API Key、endpoint 配置）
- 音频编解码与内存管理
- 音频队列维护（与事件队列的 `scheduled_tick` 关联）
- 音频播放与生命周期管理（播放完毕后释放内存）
- 对话语音的就绪等待与延迟显示逻辑

StoryMaker 不关心 TTS 的具体实现，只提供数据（`narration_text`）和钩子。

## 4. 边界情况处理总览

| 边界情况           | 处理方案                                                              | 优先级      |
| -------------- | ----------------------------------------------------------------- | -------- |
| LLM 响应延迟       | TCP 风格重传 + 缓冲期容错                                                  | 核心已覆盖    |
| 缓冲耗尽仍无回复       | 弹窗 → 重试/放弃；永久降级后可通过管理面板恢复                                         | 核心已覆盖    |
| 永久降级后玩家恢复网络    | 管理面板"恢复连接"按钮清除标记并重新请求                                             | 中        |
| 玩家暂停游戏         | 计时器冻结                                                             | 高        |
| 玩家存档           | ExposeData 序列化 ack + 队列；contextVersion 递增取消飞行请求                   | 高        |
| 玩家读档           | 恢复状态、过滤过期事件；校验 contextVersion 丢弃陈旧回调                              | 高        |
| 读档到更早的存档       | ack 回退，自动触发新请求（多消耗 token 但功能正确）                                   | 低        |
| LLM 返回格式错误     | 四层 Guard + 修正重试                                                   | 高        |
| LLM 连续返回空事件    | Empty Plan Guard 计数 + 强制要求                                        | 中        |
| LLM 返回不支持的事件类型 | Event Guard 过滤 + 日志警告                                             | 中        |
| LLM 事件执行失败     | 记录失败原因 → 下周期 deviation_report 反馈 LLM                              | 中        |
| 近期事件 Letter 爆发 | 维护事件栈，仅提取关键事件而非逐条 Letter                                          | Phase 1  |
| HTTP 401/404   | 不重传，弹窗提示配置错误                                                      | 高        |
| HTTP 429/5xx   | HTTP 层冷却重试 1 次 → 失败则触发 TCP 层重传                                    | 高        |
| 玩家切换语言         | 系统提示词自动匹配语言；未覆盖则默认英文                                              | 中        |
| 玩家关闭游戏时有飞行中的请求 | 直接丢弃；contextVersion 机制额外保护                                        | 低        |
| 连续窗口失败         | 每个窗口独立降级                                                          | 低        |
| 保存时崩溃          | 数据错误遍及整个存档，非 mod 特有问题                                             | 低        |
| 玩家与叙事者对话       | 独立轻量请求通道，即时事件绕过队列，有安全和频率限制                                        | Phase 5+ |
| 注入提示词过长        | 限制每条注入 ≤200 token，在 PromptInjector 中截断报警                          | 中        |
| 玩家个性化描述过长      | UI 提示建议字数；超出则任由 token 消耗增加（玩家有心理预期）                               | 低        |
| 调试日志文件过多       | 不自动清理，需用户手动管理                                                     | 低        |
| TTS 音频未就绪时事件触发 | 静默跳过语音，事件正常触发（TTS mod 负责）                                         | 中        |
| TTS 音频未就绪时对话回复 | 延迟显示对话，等音频就绪（TTS mod 负责）                                          | 中        |
| 低 token 模式     | 跳过非核心快照字段，用枚举代替字符串                                                | 中        |
| 玩家中途切换叙事者      | Harmony patch 仅对 defName=="StoryMaker" 生效，切换到其他叙事者自动停止调度；切换回来自动恢复 | 低        |
| 玩家首次使用未配置 API  | 发请求前检测配置完整性；若缺失则弹窗引导到 Mod 设置页                                     | Phase 2  |
